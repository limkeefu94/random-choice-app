# 反馈处理规则

这份文档用于整理用户反馈和 Bug，帮助判断问题是否需要立即处理、进入近期优化，或先记录观察。

## 反馈类型

- `Bug / 错误`：功能异常、页面报错、按钮无法使用、数据没有正确保存。
- `功能建议`：用户希望新增或扩展某个功能。
- `UI 不好用`：布局、按钮位置、手机端操作、视觉层级或交互体验不顺。
- `内容错误`：食物、购物、旅行、号码规则、预算、分类等内容不准确。
- `其他`：无法归入以上分类，但仍值得记录的反馈。

## 优先级规则

### P0

严重问题，1 人反馈也处理。

例：

- 网站打不开
- 登录失败
- 上传失败
- 数据丢失
- 按钮无法点击

### P1

影响多人使用，3 人以上重复反馈，进入近期优化。

例：

- 同一模式多人反馈抽取结果无法显示
- 多人反馈同一个入口找不到
- 多人反馈同一个设备尺寸下布局明显异常

### P2

体验优化，5 人以上反馈或明显影响手机体验，安排后续优化。

例：

- 手机端表单太长
- 某些说明文字不清楚
- 筛选项太多导致选择成本高

### P3

个人偏好或低影响建议，先记录。

例：

- 单个用户偏好的颜色、排序或文案
- 暂时不影响主流程的小建议
- 和当前版本目标关系较弱的扩展想法

## 处理状态

- `new`：新收到，尚未整理。
- `reviewing`：正在确认影响范围和重复次数。
- `planned`：决定开发，等待排期。
- `in_progress`：正在处理。
- `done`：已完成并上线或合并。
- `rejected`：确认不处理，并记录原因。
- `duplicate`：重复反馈，合并到已有问题。

## 每周处理流程

1. 汇总新反馈。
2. 合并重复项。
3. 统计出现次数。
4. 判断优先级。
5. 写成 Codex 任务卡。
6. 完成后更新通知。


## 不处理原则

- 违法用途。
- 鼓励赌博投注。
- 过度收集个人资料。
- 和应用主轴无关的大功能。
